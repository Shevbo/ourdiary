# Расходы по QR фискального чека (РФ)

## Контекст

На кассовом чеке РФ QR-код кодирует параметры фискального документа (часто строка вида  
`t=YYYYMMDDTHHmm&s=123.45&fn=...&i=...&fp=...&n=1` или URL с теми же query-параметрами).

**Официальное** открытое API ФНС «проверка чека ККТ» (`kkt-online.nalog.ru`) предполагает регистрацию внешнего пользователя и **мастер-токен** (см. условия на сайте ФНС).

Практичный путь для семейного веб-приложения без бюрократии ФНС — **сторонний сервис проверки чеков** с открытым HTTP API и токеном пользователя (по образцу документации **proverkacheka.com**: `POST /api/v1/check/get`, поля `token`, `qrraw`). Лимиты и условия — на стороне сервиса.

## Реализация в «Наш дневник»

1. **Клиент**: кнопка «QR чека» на экране ввода расхода → камера (`html5-qrcode`) → сырая строка QR.
2. **Сервер** `POST /api/expenses/from-receipt` с телом `{ "qrraw": "..." }`:
   - разбор параметров (`src/lib/fns-qr.ts`);
   - если задан `PROVERKACHEKA_API_TOKEN` — запрос к proverkacheka, разбор JSON и извлечение позиций (`src/lib/receipt-proverkacheka.ts`);
   - иначе **fallback**: одна строка расхода на общую сумму из `s=` в QR;
   - автокатегории по названию позиции (`src/lib/receipt-category.ts`);
   - несколько записей `Expense` в одной транзакции + одно начисление рейтинга за импорт.

## Дальнейшее развитие

- Подключение **официального** API ФНС после получения мастер-токена (отдельный адаптер, те же `Expense`).
- Уточнение маппинга позиций → категории (словарь, ML, ручные правила).
- Сохранение сырого JSON чека в отдельной таблице для аудита (по согласию пользователя).
